Boolean源码分析

Boolean 类将原始类型 boolean 的值包装在一个对象中，即 boolean 类型的包装类。 Boolean 类型的对象包含一个类型为 boolean 的字段。此类还提供了将布尔值与 String 互相转换的方法，以及处理 boolean 时有用的其他常量和方法。

1. 类定义

final 表示该类不可被继承，实现了 Serializable 表示是可序列化的，实现 Comparable 接口表示是可对比大小的。

属性 serialVersionUID 在反序列化时，jvm 会将字节流状态的类中的 serialVersionUID 与本地类中的 serialVersionUID 进行比较，如果相同，则进行序列化，不相同就抛 InvalidClassException 异常。

在实现序列化接口的时候，我们一般显式的给 serialVersionUID 设置一个固定值。这样无论类后期增加成员变量还是删除成员变量，都不会发生错误。如果没有显式指定一个固定值，java 序列化机制会自动生成一个 serialVersionUID，这个自动值会受类名称、它所实现的接口、以及所有的共有的私有的和受保护的成员变量的影响，如果这些值改变，那么这个自动值也会改变，在反序列化时，便会出错。

相信当看到 true 的哈希码固定为 1231 ，false 的哈希码固定为 1237 的时候肯定会非常疑惑，为什么是固定的数字？为什么固定是这两个数字？

  • 为什么是固定的数字？

    因为 boolean 的取值只有两个，所以当然可以用固定的数字。

  • 为什么固定是这两个数字？

    1. 因为这两个数字是足够大的素数，这里同样可以选择其它两个大素数。这样的话又引出了几个问题：

      • 为什么是素数呢？

        因为素数可以减少哈希碰撞。

      • 为什么素数可以减少哈希碰撞呢？

        因为素数的公因子足够少。至于为什么公因子多会增加哈希碰撞这里就不再解释了，可以自行百度。

      • 为什么是两个足够大的素数呢，2 或者 3 不行吗？

        在计算对象的哈希码时，通常会让对象内属性的哈希码参与运算，这样如果在较大桶的哈希集中使用太小的值，则可能会导致对象分布不均。

    2. 它也可以是开发人员的一个小彩蛋

      true: 1231 => 1 + 2 + 3 + 1 = 7

      7 是欧洲传统中的幸运数字。

      false: 1237 => 1 + 2 + 3 + 7 = 13

      13 被认为是不吉利的数字。
